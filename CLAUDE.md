## Le projet

App Electron (Windows/Mac/Linux) pour analyser des listes Warhammer 40K 11e édition (Space Marines, Blood Angels, Orks). Voir [README.md](README.md) pour l'installation/build.

- [main.js](main.js) — process principal Electron (fenêtre, menu natif)
- [src/index.html](src/index.html) — toute l'app (HTML+CSS+JS vanilla, un seul fichier, ~2000 lignes). Une seule dépendance vendorisée statique : [src/vendor/chart.umd.min.js](src/vendor/chart.umd.min.js) (Chart.js 4.4.1, pas de npm).
- Bases de données internes en JS : `UDB` (unités), `WDB` (armes), `DETACHMENT_DB`, `LEADER_DB`.
- Persistance : `localStorage` (pas de `window.storage`/preload défini — le code a un fallback `window.storage` mais il n'est jamais réellement disponible dans cette app).
- **`src/` est aussi le site PWA**, déployé et live depuis le 2026-07-06 sur **https://fulgurehot.github.io/40k-Roster-Analyser/** (repo public `FulgureHot/40k-Roster-Analyser`, déploiement auto via [.github/workflows/deploy-pages.yml](.github/workflows/deploy-pages.yml) à chaque push touchant `src/**`). `src/manifest.json`, `src/service-worker.js`, `src/assets/icon.png` sont autonomes sous `src/` exprès, pour que ce dossier reste un site statique déployable sans dépendre de fichiers en dehors de lui.
- Projet maintenant un **repo git** (ne l'était pas avant le 2026-07-06). Identité git locale configurée avec l'email noreply GitHub — voir session 2026-07-06 pour le pourquoi.

## Environnement de dev — contraintes connues

- **Impossible de lancer l'Electron réel (fenêtre GUI) depuis cet environnement sandboxé** : `npm start` est intercepté et n'ouvre pas la vraie fenêtre (a fini par ouvrir Notepad lors d'un test). Donc pas de test visuel direct de l'app packagée depuis ici.
- **Solution de contournement** : un [.claude/launch.json](.claude/launch.json) sert `src/` en statique via `http-server` (config `static-preview`, port 8081), utilisable avec les outils `preview_*` (Claude Preview) pour tester l'UI dans un vrai navigateur. Ça suffit pour tout ce qui ne dépend pas des API Electron (le menu natif, les raccourcis clavier ne sont pas testables ainsi).
- **Piège observé avec les outils preview** : `location.reload()` ou plusieurs cycles `preview_stop`/`preview_start` rapprochés peuvent laisser le navigateur sur un état/document obsolète (clics qui ne font rien, `innerWidth`/`innerHeight` à 0, screenshots qui timeout). Si le comportement semble incohérent après un test, forcer une vraie navigation fraîche via `location.assign('http://localhost:8081/index.html?cb='+Date.now())` avant de retester, plutôt que de conclure trop vite à un bug applicatif. **Si `innerWidth`/`innerHeight` restent à 0 même après ça** (vu le 2026-07-06, cause non identifiée), appeler explicitement `preview_resize` (ex: 1280x800) — ça force une vraie taille de viewport et corrige le rendu (canvas Chart.js à largeur 0, etc.) même quand la navigation fraîche seule ne suffit pas.
- **Piège de test** : après un premier test qui a créé une session Live/Library, une navigation fraîche recharge l'état persisté (`localStorage`) automatiquement via `DOMContentLoaded` — `localStorage.clear()` seul ne suffit pas à repartir de zéro dans un test suivant si des variables globales (`LIVE`, `_libCache`) sont déjà peuplées en mémoire ; il faut aussi les réinitialiser explicitement (`LIVE=null`) avant de retester un scénario "à froid".
