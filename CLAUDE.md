# 40K Roster Analyzer — Notes de session

> Ce fichier est mis à jour à chaque session de travail avec Claude Code, pour permettre de reprendre le contexte facilement dans une nouvelle conversation. Ne pas le documenter comme une doc utilisateur classique — c'est un journal de bord technique. (Pratique généralisée à tous les projets — règle globale dans `~/.claude/CLAUDE.md`.)

## Le projet

App Electron (Windows/Mac/Linux) pour analyser des listes Warhammer 40K 11e édition (Space Marines, Blood Angels, Orks). Voir [README.md](README.md) pour l'installation/build.

- [main.js](main.js) — process principal Electron (fenêtre, menu natif)
- [src/index.html](src/index.html) — toute l'app (HTML+CSS+JS vanilla, un seul fichier, ~2000 lignes). Une seule dépendance vendorisée statique : [src/vendor/chart.umd.min.js](src/vendor/chart.umd.min.js) (Chart.js 4.4.1, pas de npm).
- Bases de données internes en JS : `UDB` (unités), `WDB` (armes), `DETACHMENT_DB`, `LEADER_DB`.
- Persistance : `localStorage` (pas de `window.storage`/preload défini — le code a un fallback `window.storage` mais il n'est jamais réellement disponible dans cette app).
- **`src/` est aussi le futur "site racine" PWA** (voir session 2026-07-06 ci-dessous) : `src/manifest.json`, `src/service-worker.js`, `src/assets/icon.png` sont autonomes sous `src/` exprès, pour pouvoir héberger `src/` tel quel comme site statique sans dépendre de fichiers en dehors de ce dossier.

## Environnement de dev — contraintes connues

- **Impossible de lancer l'Electron réel (fenêtre GUI) depuis cet environnement sandboxé** : `npm start` est intercepté et n'ouvre pas la vraie fenêtre (a fini par ouvrir Notepad lors d'un test). Donc pas de test visuel direct de l'app packagée depuis ici.
- **Solution de contournement** : un [.claude/launch.json](.claude/launch.json) sert `src/` en statique via `http-server` (config `static-preview`, port 8081), utilisable avec les outils `preview_*` (Claude Preview) pour tester l'UI dans un vrai navigateur. Ça suffit pour tout ce qui ne dépend pas des API Electron (le menu natif, les raccourcis clavier ne sont pas testables ainsi).
- **Piège observé avec les outils preview** : `location.reload()` ou plusieurs cycles `preview_stop`/`preview_start` rapprochés peuvent laisser le navigateur sur un état/document obsolète (clics qui ne font rien, `innerWidth`/`innerHeight` à 0, screenshots qui timeout). Si le comportement semble incohérent après un test, forcer une vraie navigation fraîche via `location.assign('http://localhost:8081/index.html?cb='+Date.now())` avant de retester, plutôt que de conclure trop vite à un bug applicatif.

### Session 2026-07-06
1. **Découverte** : l'app dépend en fait de **Chart.js v4.4.1**, vendorisé localement mais sous un nom/chemin bizarre (`src/index_files/chart.umd_38Zb.js`, `<script src="index_files/...">` en tête de `<head>`) — artefact d'un "Enregistrer la page sous / Webpage complete" fait depuis un navigateur, pas un ajout intentionnel documenté. Renommé/déplacé vers [src/vendor/chart.umd.min.js](src/vendor/chart.umd.min.js), référence mise à jour dans `<script src="vendor/chart.umd.min.js">` ([src/index.html:7](src/index.html:7)). Un graphique existait déjà avec (`chartSt`, bar chart dégâts+wounds par unité dans Statistics) — donc le projet n'est pas "zero dépendance runtime" comme documenté précédemment ; c'est une dépendance vendorisée statique (pas de npm), ça reste cohérent avec l'esprit du projet mais à ne pas oublier.
2. **Ajout de 3 graphiques** (demande utilisateur : "point 1 à 4" sur une liste de proposition d'améliorations ; le point 2 — bar chart dégâts/unité — existait déjà, donc traité comme "vérifié" plutôt que recréé) :
   - **Radar chart des 6 scores d'armée** (Firepower/Durability/Mobility/Mass/Consistency/Special) dans Statistics, à côté des barres existantes — `chartRadar`, canvas ajouté dans la grille du bloc Army Score ([src/index.html](src/index.html), fonction `buildStats()`).
   - **Bar chart d'efficacité aux points** (dégâts attendus / 100pts, trié décroissant) dans Statistics, juste après le graphique dégâts/unité existant — `chartEff`.
   - **Courbe de probabilité de kill** (line chart, rounds 1-5) dans Combat Sim, au-dessus des barres de progression textuelles existantes (conservées pour la lecture exacte) — `chartSim`, dans `runSimWithDef()`.
   - Toutes les instances Chart.js suivent le pattern existant (`chartSt`) : variable globale, `.destroy()` avant recréation à chaque re-render (évite les soucis "Canvas already in use").
   - Testé en conditions réelles (clics réels, pas juste eval) : Statistics et Combat Sim, y compris re-run de la simulation pour vérifier la recréation propre des charts. Aucune erreur console.
3. **Points 5 (comparaison radar superposée en Library) et 6 (sparkline d'évolution du score dans le temps)** : proposés mais **pas encore implémentés** — l'utilisateur veut réfléchir encore. Ne pas les construire sans nouvelle confirmation.
4. **Bug : radar chart ne s'affichait pas (mais les autres charts oui)** — signalé par l'utilisateur avec un symptôme intermittent ("ça marchait pendant que je travaillais, puis plus après restart"). Cause racine réelle (ma 1ère hypothèse — `position:relative` manquant sur le conteneur — était nécessaire mais **pas suffisante**) :
   - `buildStats()` (et donc la création de `chartSt`/`chartRadar`/`chartEff`) est appelée depuis `parseRoster()` **avant** `showTab('units')` — autrement dit ces 3 charts sont toujours créés pendant que l'onglet Statistics est encore `display:none`. Un canvas mesuré à ce moment fait 0×0, et Chart.js ne se corrige pas de façon fiable même après un `.resize()` explicite une fois l'onglet affiché (confirmé par test : `.resize()` recalcule toujours 0×0 dans ce cas — pas juste une question de CSS).
   - Correctif définitif : séparation du calcul de données et de la création des charts. `buildStats()` stocke les données dans `_statsChartData` puis appelle `renderStatsCharts()` ; `runSimWithDef()` fait pareil via `_simChartData`/`renderSimChart()`. Ces fonctions de rendu sont maintenant **ré-appelées (destroy + recreate, pas resize) depuis `showTab()`** à chaque fois que l'onglet `stats`/`sim` redevient actif ([src/index.html](src/index.html), fonctions `renderStatsCharts()`, `renderSimChart()`, `showTab()`).
   - Testé en conditions réelles : après `parseRoster()` (onglet Units actif, Statistics caché), `chartRadar` mesure bien 0×0 comme prédit ; après `showTab('stats')`, il mesure la vraie taille du conteneur (150×150 ou selon la fenêtre) et affiche les 6 valeurs. Testé aussi l'aller-retour d'onglets et Combat Sim (`runSim()`) sans erreur console.
   - **Piège à retenir** : ne jamais créer un chart Chart.js pendant que son conteneur est dans un onglet caché (`display:none`) en supposant qu'un `.resize()` ultérieur suffira — il faut détruire et recréer le chart une fois l'élément réellement visible.
5. **Objectif utilisateur : utiliser l'app hors-ligne sur Pixel 7 (Android) en boutique/tournoi**, après avoir préparé sa liste à la maison. Electron ne tourne pas sur Android → décision (plan approuvé, voir `~/.claude/plans/cuddly-orbiting-popcorn.md`) : transformer `src/` en PWA offline-first, hébergée plus tard sur GitHub Pages. Implémenté en 3 stages (le 4e — hébergement — attend une confirmation explicite avant d'initialiser git / créer un dépôt GitHub, pas encore fait) :
   - **Stage 1 (responsive)** : dédupliqué le CSS `.lib-card`/`.lib-bars`/`.sort-btn`/`.compare-table` (deux définitions qui se chevauchaient partiellement, cf ancienne section dette technique ci-dessous — supprimé). Ajouté le **premier `@media (max-width:640px)`** du fichier ([src/index.html](src/index.html), fin du `<style>`) : `.f2`/`.f3` (formulaire Combat Sim) et `.army-score-grid` (nouvelle classe, remplace l'inline-style de la grille Army Score) passent à 1 colonne, `.lib-bars` à 2 colonnes, cibles tactiles agrandies (`.tab`, `.toggle-row`, `.btn`). Testé aux largeurs 412px (Pixel 7) et 1280px via `preview_resize` + lecture directe de `getComputedStyle(...).gridTemplateColumns` (plus fiable que les screenshots, qui ont un rendu/échelle capricieux dans l'outil preview).
   - **Stage 2 (info-bulles de règles tactiles)** : nouveau `RULES_GLOSSARY` ([src/index.html](src/index.html), juste après `kwB()`/`NOTABLE_KW`) — volontairement **non exhaustif** : couvre les règles-coeur stables (FIGHTS FIRST, DEVASTATING WOUNDS, FLY, VEHICLE, DEADLY DEMISE...), pas les mots-clés de fluff/faction (IMPERIUM, CHAPLAIN, TACTICUS...) pour éviter d'inventer du texte de règle. Popover générique câblé une seule fois par délégation d'événements sur `document` : `pointerover`/`pointerout` filtrés sur `pointerType==='mouse'` (hover desktop), `click` (tap mobile + toggle desktop) — le filtre sur pointerType évite le double-déclenchement hover-puis-click que les navigateurs mobiles émettent au tap. Attributs `data-rule="..."` ajoutés aux pastilles `kwB()` et `.kw-tag`. Testé via `PointerEvent`/`MouseEvent` synthétiques (hover, tap, tap-ailleurs-ferme, changement de pastille) en desktop ET en 412px — fonctionne dans les deux cas, popover ne déborde jamais de l'écran (clampé horizontalement).

Enregistrement du SW dans `index.html` gardé par `location.protocol!=='file:'` pour ne rien tenter côté Electron (les service workers exigent un contexte sécurisé — https ou localhost — que `file://` n'est pas). Testé via le serveur `static-preview` (`http://localhost:8081` compte comme contexte sécurisé pour Chrome) : SW enregistré et activé, cache `40k-analyzer-v1` contient bien `index.html`, `manifest.json`, `vendor/chart.umd.min.js`, `assets/icon.png`. **Non testé** : comportement hors-ligne réel (mode avion) — pas d'outil pour ça ici, à valider sur le vrai Pixel 7 une fois hébergé (Stage 5 du plan).
